# 0004: Outbox Pattern による DB 更新とイベント発行の整合性

- ステータス: 採用
- 日付: 2026-07-11

## 背景

メッセージ保存(DB)と副作用(SignalR 配信・通知作成・検索インデックス更新)の整合性が必要。
「DB には保存されたが配信されない」「配信されたが保存されていない」を防ぎたい。

## 選択肢

1. **Outbox Pattern(採用)**
   - Message と OutboxMessage を同一トランザクションで保存し、ワーカーが Outbox を読んで副作用を実行する。
   - 長所: DB コミットが成功すればイベントは最終的に必ず配信される(at-least-once)。
   - 短所: 配信に僅かな遅延(ポーリング間隔)が生じる。重複配信があり得るため受信側の冪等性が必要。
2. トランザクション後に直接配信
   - 長所: 低遅延。短所: 配信失敗時にイベントが失われる。プロセスクラッシュで取りこぼす。
3. 分散トランザクション
   - 短所: SignalR / RabbitMQ はトランザクションに参加できず、そもそも成立しない。

## 決定

選択肢1を採用する。現段階のワーカーは API プロセス内の BackgroundService とし、
`FOR UPDATE SKIP LOCKED` で複数インスタンスでも同じ行を二重処理しないようにする。

## 障害時の挙動

- 配信失敗は指数バックオフでリトライし、上限(5回)超過で `DeadLetteredAt` を記録する。
  RabbitMQ 導入までは、このフラグ付きの行が Dead Letter Queue の役割を持つ。
- at-least-once 配信のため、クライアントは Message Id / NotificationId で重複を除去する
  (フロントエンドの `mergeMessages` は Id で重複排除する)。
- 通知・検索インデックス更新を実装する際も、この Outbox に Consumer を追加する形で拡張する。
  モジュール間・プロセス間の非同期連携が必要になった時点で RabbitMQ へ置き換える。
