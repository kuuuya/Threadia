# 0006: メッセージ検索は PostgreSQL 全文検索から始める

- ステータス: 採用(Search モジュールは未実装。実装時の方針)
- 日付: 2026-07-11

## 背景

メッセージ検索機能が対象機能に含まれる。専用検索エンジン(OpenSearch 等)をいつ導入するかを決める。

## 選択肢

1. **PostgreSQL 全文検索(tsvector / pg_trgm)から始める(採用)**
   - 長所: 追加コンポーネントなしで開始できる。権限フィルタ(Conversation 参加者のみ)を
     同一 DB 内の JOIN で確実に適用できる。データの正本と検索対象が一致し、同期ずれがない。
   - 短所: 日本語の形態素解析が弱い(bigram で補う)。大規模データでのランキング・性能は専用エンジンに劣る。
2. 最初から OpenSearch
   - 長所: 検索品質・スケール。短所: インデックス同期・権限フィルタの二重管理・運用コストが初期段階では過剰。

## 決定

選択肢1を採用する。ただし将来 OpenSearch へ置き換えられる境界を最初から設ける。

## 置き換え可能にするための制約

- 検索は Search モジュールの公開インターフェース経由でのみ提供し、他モジュールから検索用テーブルを直接参照しない。
- 検索インデックスの更新は Outbox イベント(message.sent / edited / deleted)の Consumer として実装する。
  正本は PostgreSQL の Messages であり、検索インデックスは再構築可能な派生データとする。
- 検索結果には権限外・削除済みメッセージを含めない。権限判定は検索エンジンではなくアプリケーション側で行う。
