# 0007: 大規模グループへの配信戦略

- ステータス: 採用
- 日付: 2026-07-11

## 背景

グループの人数が増えると、1メッセージあたりの配信・通知・既読イベントが人数分に増幅される。
どこまで即時配信し、どこから抑制・バッチ化するかを決める。

## 選択肢

1. **SignalR Group による接続中ユーザーへの即時配信 + オフラインは Pull(採用)**
   - 接続中ユーザーへは SignalR Group で即時配信する。オフラインユーザーごとのメッセージ複製
     (受信箱テーブル)は作らず、再接続時に `afterSequence` API で履歴を取得する。
   - 長所: 書き込み量がメッセージ数に比例し、参加者数に比例しない。実装が単純。
   - 短所: 未読数計算は読み取り時に行う必要がある(ReadPosition 方式で解決済み。ADR 0003)。
2. ユーザーごとの受信箱(inbox)テーブルへ複製
   - 長所: 未読管理・通知が単純。短所: 大規模グループで書き込みが爆発する(write amplification)。

## 決定

選択肢1を採用する。あわせて次の方針とする。

- 大規模グループでは Presence・既読イベントのブロードキャストを抑制する
  (現実装は既読イベントを配信していない。配信を追加する場合も人数閾値で抑制する)。
- 大量のプッシュ通知はキュー(将来は RabbitMQ)へ送りバッチ処理する。
- 配信方式の変更(例: 大規模 Group の専用ファンアウトワーカー)は負荷試験で問題が確認されてから行う。
