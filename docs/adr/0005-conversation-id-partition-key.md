# 0005: 将来の分割キーを ConversationId とする

- ステータス: 採用(将来方針。現段階では単一 PostgreSQL)
- 日付: 2026-07-11

## 背景

初期は単一 PostgreSQL で運用するが、将来のシャーディングを妨げない設計上の境界を今決めておく必要がある。

## 選択肢

1. **ConversationId を分割キーとする(採用)**
   - 長所: メッセージの取得・順序保証・既読位置はすべて Conversation 単位で完結しており、
     ホットパス(送信・履歴取得・採番)がクロスパーティショントランザクションにならない。
   - 短所: 巨大な単一 Conversation がホットパーティションになり得る。
     ワークスペース横断の集計(未読一覧など)は複数パーティションへのファンアウトになる。
2. WorkspaceId を分割キーとする
   - 長所: ワークスペース内の集計が1パーティションで完結する。テナント分離が自然。
   - 短所: 巨大ワークスペース全体がホットスポットになり、再分割が難しい。
     Conversation 単位の順序保証はどのみち Conversation 内で完結するため利点が薄い。

## 決定

選択肢1を採用する。順序保証・取得の境界(= Conversation)と分割の境界を一致させる。

## 現段階での具体的な影響

- Messaging のテーブル(Messages / ConversationSequences / ReadPositions)はすべて ConversationId を
  キー先頭またはキーに含めており、分割時にトランザクション境界を跨がない。
- モジュール間で DbContext を共有せず、所属確認は公開インターフェース経由のため、
  Messaging スキーマの物理分離が API を変えずに行える。
- 実測なしにシャーディングしない。大規模 Conversation のホットパーティション対策
  (配信抑制・アーカイブ)は ADR 0007 と合わせて検討する。
