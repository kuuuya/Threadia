# 0009: 添付ファイルは署名付き URL で S3 互換ストレージへ直接アップロードする

- ステータス: 採用
- 日付: 2026-07-11

## 背景

ファイル添付機能でファイル本体の転送経路とメタデータの整合性を決める必要がある。

## 選択肢

1. **署名付き URL によるクライアント直接アップロード(採用)**
   - API はメタデータ登録と署名付き PUT URL の発行のみを行い、本体はクライアントがストレージへ直接送る。
   - 長所: ファイル本体が API サーバーを経由せず、メモリ・帯域を消費しない。大きなファイルでもスケールする。
   - 短所: 「アップロード完了」を API が直接観測できず、検証が非同期になる。
2. API 経由のアップロード(multipart)
   - 長所: 完了とバリデーションが同期的。短所: API がファイル転送のボトルネックになる。

## 決定

選択肢1を採用する。フローと整合性の担保:

1. `POST /conversations/{id}/attachments` — サイズ・拡張子・MIME・ファイル名を検証し、
   Attachment 行(MessageId=null)と署名付き PUT URL(15分)を発行
2. クライアントがストレージへ直接 PUT
3. メッセージ送信時に添付を検証(本人・同一会話・未関連付け・**ストレージ上の実在と実サイズ**)し、
   コミット後に MessageId を関連付ける
4. ダウンロードは会話参加者のみに短期(5分)の署名付き GET URL を発行する

- 完了前(未アップロード)の Attachment は Message へ関連付けない
- 関連付けられないまま24時間経過した行とオブジェクトは Workers の定期掃除で削除する
- Message 削除時は message.deleted イベントの Consumer が添付を非同期削除する
- ストレージ境界は `IObjectStorage` に抽象化し、テストはインメモリ実装を使う
